Appendix B · Client SDK: Technical Detail

What the SDK captures
and how it works.

A lightweight JS snippet. Three hooks into the WebMCP spec:

1. Wraps navigator.modelContext
Intercepts registerTool() and unregisterTool() to log which tools are live on each page, when registered, schemas. No changes to brand's existing code.
2. Hooks into SubmitEvent.agentInvoked
Boolean on form submissions distinguishing human from agent. SDK logs every agent-triggered submission with timestamp, tool name, inputs.
3. Monitors execute() callbacks
Wraps execute to measure latency, success/failure, response shape. Captures agent errors. All telemetry beaconed to Profound.

What the data unlocks beyond headless scan:

Real agent traffic patterns
Which AI agents visit (GPTBot, ClaudeBot, Gemini). Tasks attempted. Time-of-day. Frequency by platform. Ground truth.
Failure point analysis

Appendix B2 · WebMCP: Two Ways to Register Tools

Two APIs.
Same result, different approach.

WebMCP gives brands two ways to expose tools to AI agents. Both tell agents what actions are available — but they suit different situations.

Imperative API — write it in JavaScript
You define the tool entirely in code
The brand's developer writes a JavaScript function that runs when an agent calls the tool. Full control — you decide what data to fetch, how to handle errors, and what to return. The agent gets a structured result back.
navigator.modelContext.registerTool({ name: "search_products", description: "Find items by keyword and price", inputSchema: { query, maxPrice }, execute: async (input) => { const results = await fetch(`/api/search`) return { content: [{ text: results }] } } })
Best for: product search, cart actions, dynamic lookups — anything requiring a real API call or custom logic.
Declarative API — annotate your HTML
You add attributes to an existing form
No JavaScript needed. The brand adds a few attributes to an HTML form that already exists on their site. The browser reads the form fields and tells agents what parameters are available. The agent fills the form; the user still clicks Submit to confirm — keeping the human in control.
<form toolname="checkout" tooldescription="Place an order for the user"> <input name="address" toolparamdescription="Shipping address"/> <button type="submit">Place Order</button> </form>
Best for: checkout forms, search bars, sign-up flows — any existing HTML form with minimal engineering effort.
The key difference: Imperative gives full programmatic control. Declarative requires zero JavaScript — just HTML attributes on existing forms. Most brands will use both: declarative for forms, imperative for dynamic actions like product search or cart management.

Appendix D · WebMCP Security Landscape

The spec's security section
is empty.

Sections 5 and 6 of the W3C Draft contain zero content. The threats are real and documented.

Cross-tab tool chaining ("Deadly Triad")
Agent with access to banking, email, social tabs. Each call is legitimate; chaining creates exfiltration. Flagged by W3C member Tom Jones.
Tool Poisoning (Invariant Labs)
Malicious instructions in tool descriptions. Invisible to users, visible to AI. Demonstrated: SSH key exfiltration from Cursor IDE.
Tool Shadowing and Rug Pulls
Registering tools with trusted names to intercept calls. Descriptions can change after user approval. Simon Willison documented both.
destructiveHint is advisory only
Not enforced by the browser. A tool marked destructiveHint: false can still delete data. No enforcement mechanism.
Risk and opportunity. Profound can define security best practices before the W3C does. EU AI Act (Aug 2) creates a hard deadline for auditable agent interaction logs.
